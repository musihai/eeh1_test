# 严格原始口径 + 分段长度惩罚 修改方案

## 1. 方案目标

基于目前的代码与调试现象，当前主问题已经明确为：

1. **Turn 3 长度控制没有学稳**
2. 旧问题“94/95 near-miss”被压下去了，但新问题变成：
   - **over-generation（如 585!=96）**
   - **missing_answer_close_tag**
3. `fallback_normalized` 会掩盖原始生成质量，容易误导判断
4. 当前 reward 对长度错误过于一刀切，不利于学习“95 比 585 更接近正确”

因此，本方案的总体目标是：

> **删除 `fallback_normalized`，统一使用 `strict_raw` 作为训练主口径；**
> **reward 对少写和多写都罚，且偏差越大罚越重；**
> **流程层只做弱过滤，不提前替 reward 决定“长度是否合格”；**
> **Turn 3 提示从“别太早停”切换成“到 96 立刻停”。**

---

## 2. 总体修改原则

### 原则 1：删除 `fallback_normalized`
原因：
- 它会把原始生成错误“修补”成看起来不错的结果
- 导致 reward 统计看起来变好，但模型真实生成能力并没有同步变好
- 容易掩盖：
  - over-generation
  - close tag 缺失
  - 原始长度偏差

### 原则 2：只保留 `strict_raw`
所有训练主口径、主 debug 口径都基于原始 Turn 3 输出：

- 原始 `pred_len`
- 原始 `has_answer_open`
- 原始 `has_answer_close`
- 原始 `was_clipped`
- 原始 `format_score`
- 原始 `final_score`

### 原则 3：长度错误不再一刀切
不能继续用：

- `pred_len != 96 -> -1`

因为这样会导致：
- 95 和 585 一样惨
- 模型学不到“接近正确”的连续信号

### 原则 4：流程层不负责判断“长度是否正确”
流程层只负责：
- 这是不是一个“像预测答案”的结构化输出

不要负责：
- 是否恰好 96 行
- 是否接近 96

这些应该交给 reward。

## 3. 代码级修改方案

## 3.1 修改 `recipe/time_series_forecast/time_series_forecast_agent_flow.py`

### 3.1.1 删除或停用 `fallback_normalized`
如果当前代码里还保留：
- near-complete fallback
- fallback_triggered
- fallback 后再拿去算主 reward

建议：

#### 训练主路径
- 完全删除 fallback_normalized 的参与
- 不再让 fallback 影响训练 reward 或主验证指标

#### 调试辅助
如果你仍想分析“前 96 行本身好不好”，可以保留一个单独的分析函数，但它不能进入训练主路径。

---

### 3.1.2 修改 `_looks_like_forecast_answer()`：改成弱过滤器
#### 当前不建议
不要把它改成：
- `len(values) == 96`
- `len(values) >= 94`

因为那样会提前拦掉很多样本，使 reward 中“分段长度惩罚”的连续信号失效。

#### 推荐改法
让它只负责判断：
- 这是不是一个可解析的数值型答案

例如：

```python
def _looks_like_forecast_answer(self, answer_text: str) -> bool:
    values = extract_values_from_time_series_string(answer_text)
    return len(values) >= 8
```

你也可以用 4、8、12 这类较小门槛，但不要再用 94 或 96。

#### 为什么
因为你希望 reward 去表达：
- 少于 96 罚
- 多于 96 也罚
- 偏差越大罚越重

如果流程层先把这些样本都挡掉，reward 就学不到这个结构化长度控制信号。

---

### 3.1.3 `_extract_final_answer()` 只做严格提取，不做训练口径修补
建议让它：

#### 严格做的事
- 提取 `<answer>...</answer>`
- 记录失败原因：
  - `missing_answer_open_tag`
  - `missing_answer_close_tag`
  - `no_numeric_values`

#### 不要做的事
- 不要把未闭合 `<answer>` 修成正常答案后再参与训练
- 不要把超长 answer 截成 96 行后再当训练主答案
- 不要用 fallback 改变 reward 主口径

#### 原因
否则你会继续出现：
- 原始 Turn 3 很差
- reward 看起来却很好

---

### 3.1.4 保留严格原始调试字段
建议在 Turn 3 debug 中固定保留：

- `raw_pred_len`
- `has_answer_open`
- `has_answer_close`
- `was_clipped`
- `failure_reason`
- `raw_text_tail`

如果你还想做分析型截断视图，可以额外保存：
- `analysis_truncated_pred_len`
- `analysis_truncated_mse`

但不要让它进入训练主 reward。

---

## 3.2 修改 `recipe/time_series_forecast/reward.py`

这是本次方案最核心的改动。

### 3.2.1 保留格式强惩罚
格式错误仍然要强罚。

#### 建议
- 没 `<answer>`、解析失败：`format_score = -1.0`
- 正常提取到 `<answer>` 且有数值：`format_score = 0.0`

这部分不要放松。

---

### 3.2.2 MSE 继续保留，但只比较有效范围
建议：

- `pred_len < 96`：比较前 `pred_len`
- `pred_len > 96`：比较前 `96`
- `pred_len == 96`：正常比较

也就是：

```python
compare_len = min(pred_len, gt_len)
```

#### 为什么
这样可以让预测内容本身继续给训练提供信号，而不是长度错了就完全浪费掉样本。

---

### 3.2.3 长度惩罚改成“分段惩罚”
建议新增：

```python
len_gap = abs(pred_len - gt_len)

if len_gap == 0:
    length_penalty = 0.0
elif len_gap <= 2:
    length_penalty = 0.05
elif len_gap <= 5:
    length_penalty = 0.10
elif len_gap <= 20:
    length_penalty = 0.20
elif len_gap <= 100:
    length_penalty = 0.30
else:
    length_penalty = 0.40
```

然后对 over-generation 增加额外罚项：

```python
if pred_len > gt_len:
    length_penalty += min((pred_len - gt_len) / gt_len, 1.0) * 0.1
```

#### 这样设计的意义
- 95：轻罚
- 94：稍重一点
- 100：中罚
- 585：重罚

这比以前 `!=96 -> -1` 更合理。

---

### 3.2.4 最终分数结构
建议改成：

```python
total_score = format_score + mse_score - length_penalty
```

#### 注意
不要把 length_penalty 做得太大，以免完全压过 MSE。  
建议你先让：
- `length_penalty` 上限在 `0.4 ~ 0.5`
- `mse_score` 保持当前主导地位

这样模型既会重视长度，又不会完全忽略数值预测质量。

---

### 3.2.5 在 reward debug 中新增字段
建议保留：

- `raw_pred_len`
- `len_gap`
- `length_penalty`
- `under_generation`
- `over_generation`
- `exact_generation`

这样后面你就能直接看出：
- 模型现在主要是少写
- 还是多写
- 还是已经在向 96 收敛

---

## 3.3 修改 `recipe/time_series_forecast/prompts.py`

### 3.3.1 保留的内容
继续保留：

- exactly 96 values
- one float per line
- no timestamps

这些要求是对的。

### 3.3.2 调整提示重点
当前不要再继续强化：

- “不要停在 94/95”

因为你已经看到，这会把问题推向 over-generation。

#### 改成强调
- Stop immediately after the 96th value.
- Do not generate a 97th value.
- After the 96th value, close `</answer>`.

### 3.3.3 压短 `<think>`
建议 `<think>` 压缩成：
- 1 句
- 不要冗长解释

原因：
- `<think>` 太长会抢占答案生成预算
- 会放大 close tag 缺失问题

## 3.4 修改 README

README 需要同步为当前真实口径：

### 必须明确写清楚
- 训练主口径只使用 `strict_raw`
- `fallback_normalized` 不再作为训练主口径
- 最终 `<answer>` 为：

  - 无时间戳

### 额外建议补充
说明：

## 4. 不建议做的事情

### 4.1 不建议把“96 行硬停止”直接塞进训练主逻辑
因为这样太工程化，会削弱“模型自己学会停在 96”的能力。

#### 可以做什么
你可以保留一个分析工具：
- 截前 96 行
- 看前 96 行质量

但这个不要参与训练主 reward。

---

### 4.2 不建议让 `_looks_like_forecast_answer()` 变成严格 96 门槛
因为这会提前拦掉很多样本，导致 reward 中长度惩罚的学习信号传递不下去。

---

### 4.3 不建议继续使用 fallback_normalized
因为它非常容易把问题“洗平”。

---

## 5. 修改后的验证方法

修改后，不要先看总 reward，先看以下三组指标。

### 5.1 原始长度分布
统计：

- `pred_len < 96`
- `pred_len == 96`
- `pred_len > 96`

#### 目标
- `pred_len == 96` 上升
- `pred_len > 96` 明显下降
- 585 这种主失败簇消失或显著减少

---

### 5.2 失败原因分布
统计：

- `missing_answer_open_tag`
- `missing_answer_close_tag`
- `length_mismatch`

#### 目标
- `missing_answer_close_tag` 下降
- over-generation 失败下降

---

### 5.3 reward 组成项
统计：

- `format_score`
- `mse_score`
- `length_penalty`
- `final_score`

#### 目标
看到模型逐步表现为：
- `length_penalty` 降低
- `mse_score` 提升
- `final_score` 平稳上升

---

## 6. 推荐执行顺序

### 第一轮：最核心改动
1. 删除 `fallback_normalized`
2. `_looks_like_forecast_answer()` 改成弱过滤器
3. reward 改成：
   - 格式强惩罚
   - 保留 MSE
   - 分段长度惩罚
4. prompt 改成：
   - “到 96 立刻停”
   - “不要生成第 97 个值”

然后跑一轮小规模验证。

   - `pred_len > 96`

   只用于分析，不用于训练

## 7. 一句话总结

**本次最合适的方案是：**
- 删除 `fallback_normalized`
- 只保留 `strict_raw`
- reward 改成“少写和多写都罚，偏差越大罚越重”
- 流程层只做弱过滤
- prompt 从“别太早停”切换成“到 96 立刻停”

这样既不会用工程修补掩盖问题，也能把有效的长度学习信号传递下去。
