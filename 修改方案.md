# 正式 RL 问题修改方案

## 1. 目标

本方案只解决当前已经**被日志和代码同时确认**的问题，不做额外发散修改。目标有三个：

1. 让训练目标与验证目标完全一致，避免模型学到“训练给正分、验证判失败”的行为。
2. 保持实现与 `paper.pdf` 的主线一致：三阶段 workflow、episode-level delayed reward、严格最终答案协议。
3. 保持代码简洁，只做必要修改，不继续叠加历史兼容补丁。


## 2. 论文对齐原则

根据 `paper.pdf` 的 `§3.4`、`§3.5.1`、`§3.5.2`、`Appendix C.2`，本次修改遵守以下原则：

1. 保持三阶段 agent workflow 不变：
   - Turn 1: diagnostic / planning
   - Turn 2: routing
   - Turn 3: refinement

2. 保持 RL 的 episode-level delayed reward 不变：
   - reward 在 episode 结束后统一计算
   - 不引入新的中间 shaping reward

3. 最终答案协议必须严格：
   - `<think>...</think><answer>...</answer>`
   - 固定 horizon
   - 固定 timestamp grid

4. 不把外部 heuristic planner 硬编码成强监督：
   - `required_feature_tools` 继续作为分析/调试信号
   - 不改成强 reward 约束

5. 不通过“放松验证”来掩盖训练问题：
   - validation 仍保持 strict
   - 不把 validation 改成 recoverable


## 3. 已确认的问题

### 3.1 训练 reward 与验证 reward 不一致

这个问题已经确认，不是猜测。

- 训练 rollout 的最终 reward 之前允许 `allow_recovery=True`
- validation 通过 reward manager 调 `compute_score(...)`，默认 `allow_recovery=False`

结果是：

- 训练时，大量 `missing_answer_close_tag` 的样本被恢复后仍拿正分
- 验证时，同类样本直接判 `-1`

这会直接把模型优化到错误目标上。

#### 已确认的证据

当前 run 的 debug 已经显示：

- 训练侧 `reward_compute` 中，`recovered_*` 样本数量明显多于 `strict_protocol`
- 训练侧被恢复的 `missing_answer_close_tag` 样本仍可拿到稳定正分
- 验证侧同类样本全部按 strict 规则判 `-1`

因此，这不是“可能影响”，而是已经在当前 run 中发生的真实目标错位。

### 3.2 当前验证失败的主因不是 workflow，而是 final answer 协议失败

当前 step 20 validation 的核心指标：

- `validation_reward_mean = -0.8866`
- `reward_negative_one_ratio = 0.9210`
- `final_answer_accept_ratio = 0.0790`
- `format_failure_ratio = 0.9204`
- `missing_answer_close_tag_count = 1564`

这说明当前主失败模式是：

- refinement 生成经常以 `length` 结束
- 最终没有闭合 `</answer>`

而不是：

- tool 调用没走通
- workflow 大面积拒绝

#### 已确认的证据

当前 validation 中：

- `workflow_rejected_ratio` 极低
- `missing_answer_close_tag_count` 极高
- `generation_finish_reason_distribution` 以 `length` 为主

因此，当前首要问题不是 tool/workflow 逻辑，而是 Turn 3 最终答案协议没有学稳。

### 3.3 正确答案长度本身不是主瓶颈

这个问题也已经做过离线长度检查。

使用当前 SFT warm start 的 tokenizer，对真实 step-wise SFT refinement 样本统计得到：

- refinement teacher `answer_mean ≈ 2623.75 tokens`
- `answer_max = 2624 tokens`
- `think_mean ≈ 25.75 tokens`
- `full_max ≈ 9489 tokens`

而当前 refinement response cap 是 `2944` token。

这说明：

- 一个**正确**的论文式 Turn 3 teacher answer，本身是能放进当前长度预算里的
- 当前大量 `length + missing_answer_close_tag`，更像是模型没有学会及时闭合协议，而不是“正常答案天然放不下”

### 3.4 模型还存在行为层面的次级问题

这些问题已经在 debug 中观察到，但当前判断它们**不是第一优先级根因**：

1. routing 明显偏向 `chronos2`
2. refinement 大多数时候只 copy selected forecast
3. 存在少量 refinement degraded
4. routing turn 仍有少量 `missing_answer_block`

这些现象需要继续观察，但在训练/验证 reward 目标未对齐前，不应贸然大改。


## 4. 修改范围

本次计划只做两类修改：

1. **Reward 语义对齐**
2. **调试信息补强**

本次**不做**以下修改：

- 不改 backbone
- 不改 GRPO 主算法
- 不改 routing 规则
- 不改 heuristic teacher 来源
- 不新增 reward shaping
- 不把 `required_feature_tools` 接成强约束
- 不先盲目增大 `max_tokens`
- 不重做 prompt 大重构


## 5. 具体修改方案

### 5.1 训练与验证统一使用 strict final reward

#### 目标

让训练优化目标与验证/部署目标完全一致。

#### 修改内容

在 Turn 3 的训练终局 reward 计算路径中，统一使用 strict protocol：

- 训练不再对 malformed final answer 做 recoverable 正向打分
- `missing_answer_close_tag`、`invalid_answer_shape` 等协议失败，训练与验证语义一致

#### 涉及文件

- `recipe/time_series_forecast/time_series_forecast_agent_flow.py`

#### 设计理由

这最符合论文主线：

- 论文强调 episode 最终结果的 reward
- 最终结果必须是合规最终答案
- 不应让训练奖励一种验证时会被直接拒绝的输出

#### 预期直接效果

这一步的直接目标不是“立刻让模型学好”，而是先消除一个已经确认的错误优化目标：

- 训练不再继续强化 `missing_answer_close_tag` 这类 malformed final answer
- validation 的差结果不再被 train/eval mismatch 人为放大
- 后续看到的格式失败，才是模型真实还没学会，而不是 reward 语义错位造成的假象

#### 不做的事

- 不把 validation 放宽为 recoverable
- 不保留“双轨训练目标”


### 5.2 recovery 保留为调试能力，不保留为训练正反馈

#### 目标

保留对近失误样本的诊断能力，但不继续把它当成训练成功样本。

#### 修改内容

`compute_score` 仍保留 `allow_recovery=True` 能力，但用途只限于：

- debug 对照
- 失败样本分析
- 离线 inspection

训练优化使用 strict score，validation 也使用 strict score。

#### 涉及文件

- `recipe/time_series_forecast/reward.py`
- `recipe/time_series_forecast/time_series_forecast_agent_flow.py`

#### 设计理由

这样既能保留“这个样本离成功只差一个闭合标签”的可解释性，
又不把错误协议样本继续当成正样本优化。

#### 为什么这一步比“直接加大 max_tokens”更优先

当前已有证据表明：

- teacher 正确答案可以放进当前 token budget
- 当前模型大量失败的根因是没闭合协议
- 如果先扩 token，只会让 malformed 输出更长，不会自动让 reward 更一致

因此，先修 reward 目标是更符合论文、也更干净的顺序。


### 5.3 调试信息补强为“strict vs recovered 对照视图”

#### 目标

下一次出问题时，能够直接回答：

- 这个样本 strict 是否失败
- recovered 是否能恢复
- 两者分差是多少
- 是 `<think>` 太长，还是 `<answer>` 没闭合，还是写超行数

#### 修改内容

在现有 debug 结构上，只增加最关键字段，不扩散：

1. reward 级对照字段
   - `strict_score`
   - `recovered_score`
   - `recovery_gap`
   - `strict_parse_mode`
   - `recovered_parse_mode`

2. refinement 长度剖面字段
   - `answer_line_count`
   - `expected_answer_line_count`
   - `think_token_len`
   - `answer_token_len`
   - `wrote_expected_rows_before_stop`

3. run 级标识字段
   - `global_step`
   - `validate`
   - `run_name`

#### 涉及文件

- `recipe/time_series_forecast/reward.py`
- `recipe/time_series_forecast/time_series_forecast_agent_flow.py`
- `arft/trainer_validation_support.py`
- `verl/verl/experimental/reward_loop/reward_manager/naive.py`

#### 设计理由

这部分不是“加更多日志”，而是把当前最缺的诊断信息补齐。
只增加能直接定位问题的字段，避免无效噪声。

#### 为什么必须补这一层

如果不把 strict/recovered 两条分数并排记录，后面即使 validation 改善，也无法回答：

- 是模型真的学会了 strict 协议
- 还是只是 recovered 样本变少了
- 还是其它行为问题在掩盖格式问题


## 6. 暂不修改的内容与原因

### 6.1 暂不直接改 routing collapse

原因：

- 当前最致命的问题是 reward 目标不一致
- routing 塌缩很可能是这个问题的下游现象

如果在 reward 未对齐时就开始改 routing，只会同时引入多个变量。

### 6.2 暂不把 required diagnostics 接成硬约束

原因：

- 论文主线是 agent 自主 planning，不是执行外部 heuristic checklist
- 这会让实现偏离论文

### 6.3 暂不直接增大 refinement max_tokens

原因：

- 当前正确答案本身是能放下的
- 问题更像是模型没有学会及时闭合，而不是天生空间不足

如果先扩 token，只会增加序列长度和训练成本，不一定解决主因。


## 7. 实施顺序

严格按下面顺序做，不并行插入额外改动：

1. 统一训练/验证 strict reward
2. 补充 strict vs recovered debug 字段
3. 跑单元测试
4. 跑短 RL smoke
5. 检查 smoke 的 train/debug/validation 输出
6. 只有确认正常后，再重开正式 RL


## 8. Smoke 验收标准

只有同时满足下面条件，才算这轮修改完成：

### 8.1 训练链路

- 无类型错误
- 无 reward 路径异常
- 无 GRPO 数值爆炸

### 8.2 调试链路

`ts_chain_debug.jsonl` / validation debug 中必须能看到：

- `strict_score`
- `recovered_score`
- `recovery_gap`
- `answer_line_count`
- `think_token_len`
- `global_step`
- `validate`

### 8.3 行为结果

短 smoke 中至少应满足：

- 训练侧不再把 `missing_answer_close_tag` 当作正优化目标
- validation 侧 `reward_negative_one_ratio` 不再由 train/eval mismatch 人为放大
- 没有新的协议级回归

### 8.4 必须通过的“根因验证测试”

这部分是本方案的关键，不通过则不能进入正式训练。

1. **训练/验证 strictness 一致性测试**
   - 构造一个缺失 `</answer>` 但其余内容正确的最小样本
   - 训练终局 reward 与 validation reward 都必须返回 strict reject
   - 不允许出现“训练正分、验证负分”

2. **recovery 仅作诊断测试**
   - 同一个 malformed 样本同时记录 strict 与 recovered 两套结果
   - strict score 仅用于训练/验证
   - recovered score 仅用于 debug 输出

3. **真实数据短 smoke**
   - 使用正式数据、真实 tokenizer、正式 prompt
   - 跑短 RL smoke
   - 检查 debug 中 `strict_score/recovered_score/recovery_gap`
   - 检查 training side 不再把 malformed answer 统计成正优化样本

4. **长度边界 sanity test**
   - 抽取真实 refinement teacher 样本
   - 验证 teacher answer token length 仍稳定低于当前 response cap
   - 如果 teacher 自身都放不下，才允许重新讨论 token budget

### 8.5 不能承诺的事情

本方案可以保证：

- 先修正已经确认的错误优化目标
- 用测试验证修复是否真的击中根因

本方案**不能在修改前就承诺**：

- 一次修改后模型一定达到最终最优效果
- 不需要后续再观察 routing/refinement 行为

但它可以保证：后续看到的问题将是**真实模型能力问题**，而不是 reward 语义错位造成的假问题。


## 9. 正式重跑要求

一旦开始正式 RL，必须遵守：

1. 不续训当前坏 run
2. 从当前最新 SFT warm start 重新开新 RL 目录
3. 保留 SwanLab
4. 保留 chain debug


## 10. 预期结果

本次修改的**直接目标**不是立刻让模型“学得很好”，而是先保证：

1. 模型优化目标正确
2. 验证结果可解释
3. debug 能定位真实剩余问题

只有这一步成立后，后续再看：

- routing 是否仍塌缩
- refinement 是否仍过度 copy
- 是否还需要进一步调 prompt / token budget / data mixture

换句话说，本次修改的标准不是“凭感觉看起来更好”，而是：

1. 根因被测试验证
2. 训练目标被纠正
3. 调试信息足以继续定位剩余真实问题


## 11. 本方案的边界

本方案不是一次性“大修所有现象”，而是先解决已经确认的核心错位：

- 训练 reward 与验证 reward 的 strictness mismatch

这是目前最小、最干净、最贴论文主线的修复路径。
