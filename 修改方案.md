# ETTh1-OT Compact 3-Turn 当前版本修改方案

## 1. 结论概述

基于你当前 GitHub 最新代码、最近几轮 `ts_chain_debug.jsonl` 调试结果，以及当前 README / `prompts.py` / `time_series_forecast_agent_flow.py` / `run_qwen3-1.7B.sh` 的配置，当前主问题已经收缩为：

1. **Turn 3 最终答案已经大体能生成出来，但 near-complete case（94/95 行）仍会被整条判死**
2. **一部分样本仍会缺少 `</answer>`，导致 `missing_answer_close_tag`**
3. **流程层对“final answer 是否成立”的判断门槛，和 reward 层的严格长度约束不一致**
4. **README 中仍残留“最终 `<answer>` 保留时间戳”的旧描述，容易和当前代码口径冲突**

因此，下一步最优先的修改方向不是继续重写 reward，也不是再优先折腾 parser，而是：

> **先把 Turn 3 的 final answer 提取与严格长度约束对齐，并增强 near-complete case 的收尾稳定性。**

---

## 2. 当前问题的根因判断

### 2.1 不是 parser 主导问题
近期 debug 已经显示：

- `arguments` 缺失问题基本解决
- tool parser 已不是当前主瓶颈

所以本轮不再优先改 parser。

---

### 2.2 不是 reward 链路主导问题
近期 debug 已经显示：

- 原始 reward 和 trainer 口径已经比前几轮清晰
- trainer 侧不再是“完全吞掉负 reward”的主问题
- 验证集 `reward_mean` 已经能稳定到 0.36~0.37 左右

说明 reward 链路已经不是第一主矛盾。

---

### 2.3 当前真正的主问题：final answer 提取层与 reward 严格长度层不一致
当前系统中存在这样的不一致：

#### 流程层
- 只要输出看起来“像一个 forecast answer”，就可能被当作 Turn 3 final answer

#### reward 层
- 必须严格 96 行，否则直接 hard fail / `-1.0`

因此会出现：

- 模型输出了完整 `<answer>...</answer>`
- 但只有 94/95 行
- 流程层认为“这已经是 final answer”
- reward 层又直接判死

这会让 RL 学到很差的信号。

---

### 2.4 另一类主问题：`missing_answer_close_tag`
近期 debug 显示，失败原因中仍有大量：

- `missing_answer_close_tag`

这说明模型在 Turn 3 收尾时，仍存在：

- 最后一小段没写完
- 或已经写到 near-complete，但没有补上 `</answer>`

---

## 3. 修改目标

本次修改目标按优先级分为三层：

### 第一层：让流程层和 reward 层标准一致
即：
- 不再让 70~80 行这种“半成品”轻易被流程层当作 final answer
- 让 final answer 的判定更接近 reward 真实要求

### 第二层：给 near-complete case 一个流程层容错
即：
- 对已有 `<answer>` 开头、且已写出大量值但缺少 `</answer>` 的样本，不在流程层直接视为“完全无答案”

### 第三层：进一步提高 Turn 3 收尾稳定性
即：
- 降低 `missing_answer_close_tag`
- 提高 `pred_len == 96`
- 降低 `pred_len in {94,95}`

---

## 4. 按文件修改方案

## 4.1 修改 `recipe/time_series_forecast/prompts.py`

### 目标
进一步强化 Turn 3 的 finishing 约束，避免模型停在 94/95 行或忘记闭合 `</answer>`。

### 你现在应保留的内容
当前 prompt 已经明确要求：
- `<answer>` 必须恰好 96 行
- 每行只允许一个 float
- 不要时间戳

这一点不要放松。

### 建议新增/强化的内容
在 Turn 3 最终答案要求处，补充更直接的 finishing 提示，例如：

- Output **exactly 96** values, not 94 or 95.
- After the **96th** value, you must immediately output `</answer>`.
- Do not stop early.
- Do not include timestamps.
- Do not include any extra explanation after `</answer>`.

### 建议写法示意
```python
# Turn 3 final answer requirements
# - <answer> must contain exactly 96 lines
# - each line must be a single float value
# - no timestamps
# - do not stop at 94 or 95 values
# - after the 96th value, immediately output </answer>
# - no extra text after </answer>
```

### 目的
不是让模型学新知识，而是显式强化“收尾动作”。

## 4.2 修改 `recipe/time_series_forecast/time_series_forecast_agent_flow.py`

这是本次修改的核心文件。

### 4.2.1 修改 `_looks_like_forecast_answer()`

#### 当前问题
当前“像 forecast answer”的门槛偏低，容易让：
- 70~80 行
- 或其他 near-but-not-valid 输出

被流程层过早认定成 final answer。

#### 修改目标
把流程层 final answer 的识别门槛，拉到更接近 reward 层。

#### 建议改法
原来的低门槛逻辑，改成更严格版本，例如：

```python
def _looks_like_forecast_answer(self, answer_text: str) -> bool:
    values = extract_values_from_time_series_string(answer_text)
    return len(values) >= self.forecast_horizon - 2
```

也就是先用：
- `>= 94`

而不是一个明显偏低的阈值。

#### 更严格的可选版本
如果你希望流程层完全和 reward 一致，也可以直接：

```python
return len(values) == self.forecast_horizon
```

#### 推荐
先用 `>= 94`，因为这样：
- 仍能保留 near-complete case 的调试空间
- 但已经足够挡掉 70~76 那种旧截断簇

---

### 4.2.2 修改 `_extract_final_answer()`

#### 当前问题
目前它只认：

```python
<answer> ... </answer>
```

如果没有闭合标签，就整段失败。

#### 修改目标
给 near-complete case 一个流程层 fallback，不要一刀切当成“没有 final answer”。

#### 建议改法
保留严格分支优先：

1. 先严格匹配 `<answer>...</answer>`
2. 如果失败，再走 fallback：

##### fallback 条件
- 有 `<answer>`
- 没有 `</answer>`
- `<answer>` 后能解析出很多值（建议 `>= 90`）

##### fallback 行为
- 取 `<answer>` 之后的所有文本作为 `candidate_answer_text`
- 交给 `_looks_like_forecast_answer()` 再判

#### 伪代码示意
```python
match = re.search(r"<answer>(.*?)</answer>", response_text, re.DOTALL)
if match:
    answer_text = match.group(1).strip()
    if self._looks_like_forecast_answer(answer_text):
        return answer_text, confidence

# fallback for near-complete unclosed answer
if "<answer>" in response_text and "</answer>" not in response_text:
    candidate = response_text.split("<answer>", 1)[1].strip()
    values = extract_values_from_time_series_string(candidate)
    if len(values) >= self.forecast_horizon - 6:   # 例如先用 >=90
        return candidate, fallback_confidence
```

#### 注意
这不是要放宽正式评测，而是为了：
- 区分“根本没进 final answer”
- 和“已经写了 95 行只是缺闭合标签”

最终 reward 仍然可以保持严格 96 行。

## 5. reward.py 现在怎么处理

### 结论
**当前不要优先重写 reward 主体。**

原因：
- reward 已经足够严格
- 问题已经明确不是 reward 本身坏了
- 现在主要卡在 final answer 收尾与流程层识别

### 可以做的唯一补充
新增一个**分析型** near-miss 长度分，不参与正式训练，只用于 debug：

- `pred_len == 96` -> 1.0
- `pred_len == 95` -> 0.95
- `pred_len == 94` -> 0.9
- 其余再低

### 用途
帮助你判断：
- 当前模型是在向 96 收敛
- 还是在随机波动

注意：
- 不建议直接替换正式 reward
- 只作为分析指标附加到 debug log 里
