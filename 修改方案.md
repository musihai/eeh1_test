## 保留的必要差异

### 1. ETTh1 任务协议解析

作者原始流程默认按通用时序文本解析；本分支的 RL 样本包含任务头和 `OT=...` 形式的历史值，因此保留了一个额外协议层：

- `recipe/time_series_forecast/task_protocol.py`

它只做三件事：

- 从原始 prompt 中拆出 `Historical Data`
- 解析 `Target Column / Lookback / Horizon / Data Source`
- 兼容 `timestamp OT=value` 与 `timestamp value`

这是本分支唯一明确多出来、且无法删除的运行时适配层。

### 2. ETTh1 单变量 prompt

`recipe/time_series_forecast/prompts.py` 已改成：

- 数据集描述为 `ETTh1`
- 目标列描述为 `OT`
- 明确是单变量预测
- 仍保留作者原始的 3-turn runtime 结构：
  - Turn 1 特征
  - Turn 2 预测
  - Turn 3 `<think><answer>`

### 3. ETTh1 单变量模型配置

本地工具模型配置保持单变量：

- `recipe/time_series_forecast/models/patchtst/config.json`
- `recipe/time_series_forecast/models/itransformer/config.json`


## 当前剩余的不一致

从代码和数据角度看，目前只剩下两类“有意保留”的不一致：

1. 作者源码没有 `task_protocol.py`，本分支保留它来兼容 ETTh1 RL 样本协议。
2. 作者原始任务覆盖更多数据集，本分支只保留 ETTh1 单变量 `OT` 所需的数据和配置。

除此之外，训练入口、agent flow、model server、工具调用路径都已经尽量回收到作者原始结构。

## 已补齐的两阶段训练入口

当前仓库已经补上：

- ETTh1 单变量 `OT` 的 multi-turn SFT 数据生成脚本  
  `recipe/time_series_forecast/build_etth1_sft_dataset.py`
- ETTh1 单变量 `OT` 的 SFT 训练脚本  
  `examples/time_series_forecast/run_qwen3-1.7B_sft.sh`
- `SFT checkpoint -> RL` 的衔接方式  
  使用 `global_step_x/huggingface/` 作为后续 RL 的 `MODEL_PATH`

## 当前剩余的实现差异

当前默认生成的 SFT 数据为了稳定和生成成本，使用的是 `ground_truth` 模式的 Turn 2 预测候选。这保证了：

- 可以不依赖模型服务直接生成 parquet
- 可以先把 `SFT -> RL` 工程链路完整打通

如果要继续向“作者式 teacher tool 轨迹”靠拢，后续可以再用：

```bash
python recipe/time_series_forecast/build_etth1_sft_dataset.py \
  --prediction-mode reference_teacher
```

重新生成一版由参考专家模型提供 Turn 2 预测的 SFT 数据。
