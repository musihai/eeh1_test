# 基于 ts_chain_debug.jsonl 的问题定位报告与修改顺序

## 1. 结论概览

基于 `debug_logs/ts_chain_debug.jsonl` 的分析，目前可以明确锁定：

1. **主问题不是 reward 公式本身，而是最终答案格式通过率太低**
2. **即使格式通过，输出长度也经常不等于 96**
3. **原始 reward 中确实存在负分（例如 `-1.0`），但进入 trainer 的 reward 统计里基本只剩非负值**
4. **tool parser 的 `arguments` 缺失问题确实存在，但频率很低，不是当前第一主因**

因此，当前系统最应该优先修的不是“更复杂的 reward 设计”，而是：

- 先让 Turn 3 的最终 `<answer>...</answer>` 稳定通过
- 再让输出长度稳定为 96
- 再查清楚为什么 trainer 侧没有看到负 reward

---

## 2. 已确认的问题与证据

## 2.1 最终答案格式失败率很高（主问题）

在 `reward_compute` / `reward_manager_output` 相关记录中，可以看到：

- `format_score = -1.0` 的样本数量明显多于 `format_score = 0.0`
- 中位数 reward 直接落在 `-1.0`
- 验证集聚合统计中：
  - `reward_mean` 为显著负值
  - `format_score_mean` 同样显著为负

这说明：

> 大量样本在 reward 的第一步格式检查就直接失败了，根本还没进入“数值预测质量比较”的阶段。

### 直接含义
当前你训练卡住，不是因为模型“只是预测得不够准”，而是：

- 很多样本连合法的最终答案格式都没产出
- RL 的大部分学习信号被格式问题吞掉了

---

## 2.2 即使格式通过，长度也经常不对

对 `format_score == 0.0` 的样本进一步观察，`pred_len` 分布里最常见的并不是 96，而是：

- 0
- 70~75 左右
- 50 多
- 只有极少数样本正好等于 96

这说明当前模型在 Turn 3 还没有真正学稳：

- 合法 XML 包装
- 恰好 96 个预测值

也就是说，当前不是“格式与长度都稳定，数值还差一点”，而是：

> **最终输出协议本身还没有学稳。**

---

## 2.3 原始 reward 有负分，但 trainer 看到的 reward 几乎都非负

从你之前贴的 `reward.py` 逻辑看：

- 格式错误时，应直接返回 `-1.0`
- 这是一个强惩罚

而在 `ts_chain_debug.jsonl` 中，确实也能看到原始 reward 里有：

- `reward_score = -1.0`
- `reward_min = -1.0`

但是你训练日志和 `trainer_reward_input` 相关统计里经常看到：

- `reward_min = 0.0`
- `has_negative = False`

这说明：

> 原始 reward 的负值并没有完整地进入 trainer 的策略更新链路。

### 这意味着什么
这通常会严重削弱 RL 的区分能力：

- 格式错样本本来应该受到强惩罚
- 但 trainer 看到的可能只是 0
- 最终导致“差样本”和“极差样本”之间分不太开

这是第二个关键问题。

---

## 2.4 Tool parser 的 `arguments` 缺失问题存在，但不是当前主因

在 `tool_parser_extract_one` 相关记录里：

- 大多数 tool call 都带了 `arguments`
- 只有极少数无参工具调用只输出了 `{"name": ...}`

这说明：

- 你之前的判断是对的：模型和 parser 的无参工具格式约定不一致
- 但从占比看，它只是一个**次要问题**

### 目前判断
这个问题值得修，但它不是“reward 卡在低位”的第一主因。  
当前最关键的瓶颈仍是：

- 最终答案格式通过率
- 最终答案长度正确率
- reward 负值在 trainer 里的消失

---

## 3. 为什么你看到的训练 reward 曲线会“看起来不太动”

这部分也已经可以解释清楚。

## 3.1 你当前 reward 本身动态范围就比较钝
你现在的严格 ablation reward 只保留：

- 格式
- 长度
- MSE

而且 MSE 项用了：

- `1 / (1 + log(1 + mse))`

这个设计的优点是稳定，但会带来两个问题：

1. 差样本之间分数差异被压缩
2. 只要长度正确，就有一个固定的 `+0.1` 保底

因此，很多“格式过关但预测很一般”的样本，会天然堆在：

- `0.1 ~ 0.15`
- 或 `0.3 ~ 0.5`

所以你之前看到：

- `critic/rewards/max ≈ 0.45 ~ 0.5`
- `critic/rewards/mean ≈ 0.1 ~ 0.15`

并不奇怪。

---

## 3.2 但真正更大的问题是：大多数样本根本没进到数值比较阶段
由于格式失败率高，真正进入 `length_score + mse_score` 的样本比例不够高。

因此训练过程中“真实可学习信号”被大幅稀释，导致：

- reward 曲线不容易显著上升
- 即使有进步，也被大量格式失败样本掩盖

---

## 4. 当前最优先的修改顺序

下面给出建议的修复顺序。原则是：

> 先修最基础的最终答案协议，再修 trainer reward 链路，最后再修 reward 细节和 parser 容错。

---

## 第一步：先修 Turn 3 最终答案协议（最高优先级）

### 目标
让模型更稳定地产生：

```xml
<think>...</think>
<answer>
96 lines of float
</answer>
```

### 要做的事
1. **SFT 样本中，Turn 3 的目标格式必须 100% 纯净**
   - 不允许混：
     - 裸数字
     - 只有 `<answer>`
     - 缺失闭合标签
     - 多余文字
   - Turn 3 统一只保留一种格式

2. **Prompt 里强化“必须输出恰好 96 行”**
   - 不只是“请输出”
   - 而是明确写：
     - exactly 96 values
     - one per line
     - no extra text

3. **不要再让中间阶段混用 XML 与非 XML**
   - compact 3-turn 已经比你以前的 candidate/refinement 多阶段稳定得多
   - 现在要继续坚持这个收缩方向

### 你现在最需要补看的调试信息
从 `ts_chain_debug.jsonl` 中进一步筛出：

- `format_score = -1.0` 的样本 10 条
- 看它们具体失败在：
  - 没有 `<answer>`
  - `<answer>` 没闭合
  - 有 `<answer>` 但解析不到值
  - 文本被截断

---

## 第二步：把长度 96 从“奖励项”改成“硬约束”

### 当前问题
现在长度只要够了就给 `+0.1`，不够按比例给分。  
这会让很多“70 多个值”的输出也能拿到一点分。

### 建议修改
在 reward 里改成：

- `pred_len != gt_len` 直接 hard fail
- 不再用“比例给长度分”

### 这样做的好处
- 让模型明确知道：  
  “不是 96 个值，就不是合法答案”
- 比现在这种“70 个值也能拿一点分”的设计更利于学稳协议

### 注意
这一步最好在第一步（格式更稳定）之后做。  
否则如果一开始就太严格，训练早期可能全军覆没。

---

## 第三步：查清楚为什么 trainer 侧没有负 reward

这是非常关键的一步。

### 当前现象
- 原始 reward 有 `-1.0`
- trainer 输入统计几乎没有负值

### 你需要排查的代码逻辑
在 reward manager 到 trainer 之间，继续搜和打印以下操作：

- `clamp`
- `relu`
- `nan_to_num`
- `max(0, reward)`
- `np.maximum(reward, 0)`

### 目标
确认到底是谁把负 reward 改成了非负值。

### 为什么重要
如果格式错本来应该 `-1.0`，最终只被当成 `0.0`，那 RL 的惩罚强度会大幅减弱。

---

## 第四步：再考虑修改 reward 细节

在格式与长度协议都更稳之后，再去改 reward 细节才有意义。

### 当前 reward 的主要问题
1. 长度分是固定保底
2. MSE 项 log 压缩较强
3. 关掉结构项后，差样本之间区分度不够

### 建议修改方向
1. **长度从奖励改为约束**
2. **保留 MSE 主项，但让差样本掉得更快**
3. **后续可考虑补一个轻量 shape penalty**
   - 方差比率
   - 均值偏移
   - 一阶差分合理性

但这些都应该放在：
- 格式稳定
- 长度稳定
- trainer reward 口径查清楚
之后再做。

---

## 第五步：顺手修 parser 对无参工具的兼容

虽然这不是当前主因，但建议顺手修掉。

### 建议改法
当 tool call 缺 `arguments`，且该工具 schema 允许空对象时：

- 自动补成 `{}`

而不是直接 KeyError。

### 收益
- 消除少量但烦人的 tool call 格式漂移
- 让 agent flow 更稳

---

## 6. 现在不要优先做的事

为了避免排查方向跑偏，以下动作不建议现在优先做：

1. **不要先重写更复杂的 reward**
   - 现在主瓶颈不是 reward 公式精度，而是格式失败率太高

2. **不要先恢复 change-point / season-trend**
   - 在最终答案协议没学稳前，加更多结构项只会让分析更复杂

3. **不要先回到 candidate/refinement 多阶段**
   - 你现在的 compact 3-turn 比之前那套更适合先把流程跑稳

---

## 7. 最终建议顺序（可直接照做）

### 阶段 A：先稳格式
1. 抽 10 条 `format_score=-1.0` 样本，定位失败类型
2. 清洗/统一 Turn 3 的 SFT 样本格式
3. 强化 Turn 3 prompt：必须 `<answer>`、必须 96 行

### 阶段 B：再稳长度
4. 看 `pred_len` 失败样本主要是少行、空输出还是截断
5. 把长度从奖励项改成更严格的合法性约束

### 阶段 C：查 trainer reward 口径
6. 打印 reward manager 输出
7. 打印 trainer 输入 reward
8. 确认负 reward 到底在哪里被吃掉

### 阶段 D：最后修 reward 细节和 parser 容错
9. 调整 MSE 项的敏感度
10. 补轻量 shape penalty
11. 修 parser 对无参工具自动补 `{}`

---

## 8. 一句话总结

基于 `ts_chain_debug.jsonl`，现在已经可以明确：

> **当前系统最核心的问题不是“预测还不够准”，而是“最终答案格式通过率太低”，导致大多数样本在 reward 第一关就被判死刑；同时 trainer 侧又没有完整看到负 reward，进一步削弱了训练信号。**

所以你的修改顺序应该是：

1. **先修 Turn 3 最终答案格式**
2. **再把长度 96 变成硬约束**
3. **再查清楚负 reward 为什么在 trainer 里消失**
4. **最后再优化 reward 细节和 parser 容错**
