# 修改方案（严格对齐 Cast-R1 论文版）

## 1. 目标

本方案以 [Cast-R1 论文 PDF](/data/linyujie/Cast-R1-TS-main/Cast-R1-TS-main/Tao%20等%20-%202026%20-%20Cast-R1%20Learning%20Tool-Augmented%20Sequential%20Decision%20Policies%20for%20Time%20Series%20Forecasting.pdf) 为主线，对齐当前实现，并同时回应当前实验中暴露出的三类问题：

1. `Turn 3` 的自由数值生成导致训练容量过多消耗在格式、长度和收尾上。
2. routing 容易塌缩到单一模型，没有学出论文中的 state-aware model selection。
3. SFT / RL 样本构造没有充分体现论文中的：
   - memory-based state management
   - stage-aware prompt
   - 单模型 routing
   - agent-side refinement
   - composite reward
   - curriculum learning

本版方案的总原则是：

- 保留论文主线：
  - `Turn 1` 诊断
  - `Turn 2` 单模型 routing
  - `Turn 3` agent 自己做 refinement 并生成最终 forecast
- 不额外给 RL 加显式 routing 监督头；
- reward 回到论文式 composite design；
- SFT / RL 样本围绕论文中的决策轨迹重新构造。


## 2. 论文主线到底是什么

## 2.1 三阶段 workflow

论文中的标准 workflow 是：

1. `Turn 1`：Feature Extraction / Diagnostic Perception
2. `Turn 2`：Model Routing
3. `Turn 3`：Refinement & Output

对齐论文后，需要明确：

- `Turn 2` 的标准动作是：根据 `Turn 1` 的分析状态，选择一个 forecasting model，并调用一次 `predict_time_series`。
- `Turn 3` 的标准动作是：在已有 `analysis history + selected model prediction` 的基础上，做 reasoning、refinement、final output generation。
- 论文没有把“多个 forecasting model 同时出 forecast，再比较/融合”写成主流程。


## 2.2 agent 根据什么信息选择模型

论文里 routing 的依据是 memory state，而不是先看多个 candidate forecast 再挑。

agent 选择 forecasting model 时，应综合以下信息：

- historical observations
- contextual signals
- data quality assessment
- global statistical characterization
- structural / dynamic analysis
- event-level summary
- residual diagnostics
- previous tool outputs
- previous actions and maintained state

也就是说，agent 学的是：

- 如何理解诊断工具提供的证据；
- 如何把这些证据组织进 memory；
- 如何基于当前 state 做 model routing；
- 如何在 selected model 的 initial forecast 基础上进一步 refine。


## 2.3 agent 怎么知道这个模型更好

论文中的答案分成两段：

1. SFT 阶段  
   通过 curated trajectories 学会：
   - 工具调用格式
   - 合理的 tool sequence
   - 基本 routing pattern
   - 基本 prediction revision pattern

2. RL 阶段  
   通过 episode-level composite reward 学会：
   - 哪种 routing 更容易得到好 forecast
   - 哪种 refinement 更容易提高最终 reward

因此，本方案保持：

- 不给 RL 加显式 routing classification loss；
- 不给 policy 直接喂 best-model label；
- routing 仍由 RL 从 composite reward 中自然学出。


## 3. Turn 3 到底由谁完成

这里必须严格对齐论文。

论文写法是：

- selected model 先产生一个 `initial forecast`
- 然后系统再做 `reasoning-based forecasting` 生成 candidate prediction
- 如果触发 revision，再做 final refinement
- 最终输出 `<answer>` 中的 final serialized forecast

所以结论是：

1. `Turn 3` 的修正不是再调一个新工具完成的。  
   论文中的 `Turn 3` 被限制为：
   - reasoning
   - refinement
   - final output generation

2. `Turn 3` 的最终 forecast 仍然是 agent 完成的。  
   不是程序完全代写 `<answer>`。

3. selected model 的 forecast 在论文中是 `initial forecast / intermediate predictive evidence`。  
   最终输出应当建立在这个 forecast 之上，但论文没有把 final answer 限制成“程序直接复制工具输出”。

因此，本方案放弃之前“程序序列化最终 `<answer>`”的表述，改为：

- final answer 仍由 agent 生成；
- 但 agent 的 final answer 必须建立在 selected model prediction 与 accumulated evidence 之上；
- refinement 是 agent-side reasoning，不是新工具调用。


## 4. 如何在不偏离论文的前提下解决当前 bug

论文没有专门处理你现在的 `94/95/72` 提前收尾问题，因此这里需要做**不改变主线的工程约束**。

## 4.1 不能做的事

以下做法与论文主线不一致，不作为主方案：

- `Turn 2` 同时调用多个 forecasting model，再比较后挑一个；
- `Turn 3` 不让 agent 生成最终 forecast，而由程序直接代写 `<answer>`；
- 把 refinement 变成新的工具调用；
- 用 strict length hard gate 让格式问题压过 forecasting quality。


## 4.2 可以做的工程稳定化手段

以下做法可以作为工程增强，但不改变论文 workflow：

1. 在 `Turn 3` 保留 agent 生成 final answer，但加入更强的 prompt 约束：
   - final answer 必须基于 selected model prediction
   - 只允许小幅修正，不允许脱离 selected forecast 大范围重写

2. 增加监控与分析字段：
   - final answer 与 selected forecast 的距离
   - refine 前后误差变化
   - refine 前后结构变化

3. 保留 stop string、parser、格式检查，但不再让长度项成为主学习信号。

4. 可以考虑在解码或后处理阶段做轻量校验：
   - 如果 final answer 明显偏离 selected forecast 且误差更差，则记为 failure case 用于后续 repair 数据；
   - 但不把 `<answer>` 完全交给程序代写。


## 5. reward 设计

## 5.1 论文中的 reward 主线

论文中的 reward 是 episode-level composite reward，包含：

1. 预测误差主项  
   - 基于 MSE
   - 做归一化与 log-transform，以稳定训练

2. 趋势与季节性一致性

3. 局部 turning point / structural alignment

4. format validity

5. output length consistency

这些 reward 项在 episode 结束后统一聚合成一个 scalar signal。


## 5.2 对当前实现的改法

reward 应回到论文式设计：

1. `prediction_error_reward`
   - 主项；
   - 使用论文式 normalized + log-transformed MSE；
   - 训练主口径按论文；
   - 同时继续记录 raw-scale MSE / MAE 作为评估和监控指标。

2. `trend_seasonality_reward`
   - 恢复。

3. `structural_alignment_reward`
   - 恢复；
   - 包括 local turning points 的容忍窗口。

4. `format_validity_reward`
   - 保留；
   - 因为 final answer 仍由 agent 生成，所以它仍然是有效训练信号。

5. `length_consistency_reward`
   - 保留；
   - 因为 final answer 仍由 agent 生成，所以它仍然是有效训练信号。


## 5.3 不再采用的做法

- 不再使用 strict length hard gate；
- 不再把长度 mismatch 直接打成主导性负分；
- 不再让格式和长度压过 prediction error；
- 不把 reward 改成“程序保证长度后长度项恒成立”。


## 6. memory 与 prompt 也要对齐论文

## 6.1 memory-based state management

论文非常强调 memory-based state management，这一点当前方案里必须明确写进去。

memory 中应包含：

- historical observations / truncated observations
- contextual signals
- tool-derived analysis summaries
- prediction results
- previous actions
- intermediate reasoning state

后续每一轮 prompt 不是简单拼接所有内容，而是基于当前阶段动态组装。


## 6.2 stage-aware prompt construction

必须保持论文里的三阶段限制：

1. `Turn 1`
   - 只能调用 feature extraction / diagnostic tools
   - 不允许 prediction tool

2. `Turn 2`
   - 已有 analysis history，但还没有 prediction result
   - 只允许 forecasting model invocation

3. `Turn 3`
   - 已有 analysis history 和 prediction results
   - 不再调用工具
   - 只允许 reasoning、refinement、final output generation

另外，论文还强调：

- 早期 turn 可给更多原始历史数据；
- 后期 turn 尤其是 `Turn 3`，应当更多依赖 memory 中的高层摘要和 prediction result，而不是重复注入长原始输入。


## 7. SFT 样本如何构造

## 7.1 SFT 的角色

论文中的 SFT 是：

- 200 个 curated high-quality samples；
- 通过 step-level supervision 让系统学会在 intermediate state 上做动作选择；
- 让系统学会：
  - 正确调用工具
  - 生成有效中间预测
  - 保持一致 state transition

因此，SFT 的目标不是教模型“自由写 96 个数”，而是教模型：

- 如何做 stage-aware decision；
- 如何在 `Turn 1` 选择合适的诊断工具；
- 如何在 `Turn 2` 做单模型 routing；
- 如何在 `Turn 3` 基于 initial forecast 做 agent-side refinement。


## 7.2 当前 SFT 的不对齐点

当前 SFT 构造的主要偏差：

1. feature tools 固定全调用  
这不符合论文里的 planning / information acquisition strategy。

2. final answer 只是在 teacher 数值文本上做模板填充  
这不够体现论文中的 reasoning-based forecasting 和 refinement。

3. 当前数据没有明确体现 “conditioned on intermediate states 的 step-level supervision”  
这也是偏离论文的一点。


## 7.3 新的 SFT 轨迹结构

每条 SFT 轨迹应包含：

1. `Turn 1`
   - 根据样本状态选择必要的 diagnostic tools；
   - 工具可并行执行，但必须是 state-dependent，而不是固定脚本。

2. `Turn 2`
   - 根据 `Turn 1` 的 analysis history 选择一个 forecasting model；
   - 调用一次 `predict_time_series`。

3. `Turn 3`
   - agent 基于：
     - analysis history
     - selected model prediction
     - memory summary
   - 自己生成 final forecast；
   - 必须体现 refinement，而不是简单复制 tool output。


## 7.4 SFT 轨迹如何离线生成

为了生成高质量 SFT 轨迹，离线阶段仍可运行全体 forecasting experts，并保存：

- 每个模型的 raw-scale `MSE / MAE`
- best model
- score margin
- forecast vectors
- diagnostic summaries

但这些信息仅用于：

- 构造合理 trajectory；
- 决定这条 sample 在 `Turn 2` 应演示哪个模型；
- 决定这条 sample 是否应在 `Turn 3` 展示 refinement。

它们不是在线 policy 可见输入。


## 7.5 SFT 样本类型

更贴近论文的 SFT 样本可按以下类型组织：

1. `Route-Only`
   - 分析后路由清晰；
   - selected forecast 已较合理；
   - `Turn 3` 做轻量反思并直接生成 final answer。

2. `Route-Then-Refine`
   - selected forecast 方向正确，但局部存在不合理波动；
   - `Turn 3` 做明确 refinement。

3. `Quality-Aware Route`
   - data quality / residual diagnostics 起关键作用；
   - agent 需要基于质量信息路由。

4. `Regime-Aware Route`
   - non-stationarity / changepoint / volatility 明显；
   - agent 需要基于 regime 信息路由并 refine。


## 7.6 SFT 样本筛选与平衡

SFT 的 200 条 curated samples 需要覆盖：

- best-model 分布不能严重失衡；
- route-only 与 route-then-refine 都要有；
- easy / medium / hard 都要有；
- regime-shift / quality-issue / residual-hard 样本不能太少。

同时，builder 需要尽量保证：

- 使用的是“reasonable sequence of tool invocation, model selection, and prediction generation”
- 而不是固定流程模板。


## 8. RL 样本如何构造

## 8.1 RL 的角色

RL 的目标是让模型在 multi-turn workflow 中学会：

- planning
- tool usage
- model invocation
- prediction revision

即：

- 何时调用哪些诊断工具；
- 如何基于 memory state 选择 forecasting model；
- 如何在 selected forecast 上做 refinement；
- 如何让整个 episode 的 final reward 更优。


## 8.2 RL prompt 设计

RL prompt 应保持与论文一致：

- 给历史窗口和任务说明；
- 通过 stage-aware prompt 动态限制动作空间；
- 不暴露 offline teacher label；
- 不暴露 best model；
- 不暴露 model_scores。

routing 必须仍由 policy 在 sequential workflow 中学出。


## 8.3 RL 的离线元数据

为了做 curriculum 与采样，离线构造时应为每个窗口补齐：

- `reference_teacher_error`
- `reference_teacher_error_band`
- `normalized_permutation_entropy`
- `normalized_permutation_entropy_band`
- `offline_best_model`
- `offline_margin`
- `quality_issue_flag`
- `difficulty_stage`

其中：

- `reference_teacher_error`
- `normalized_permutation_entropy`

是论文里的两条 difficulty 主轴。

`best_model / margin` 只保留为辅助分析与分布平衡维度。


## 8.4 RL curriculum

严格按论文，curriculum 用两维定义：

1. model-based prediction difficulty  
   由 reference teacher forecasting error 衡量。

2. data-intrinsic complexity  
   由 normalized permutation entropy 衡量。

训练过程从 easy 到 hard：

### Stage A：Easy

- teacher error 低；
- permutation entropy 低；
- 先学稳定 tool usage 和基本 routing。

### Stage B：Medium

- teacher error 升高；
- permutation entropy 中等；
- 学更复杂状态下的 routing 与 refinement。

### Stage C：Hard

- teacher error 高；
- permutation entropy 高；
- 学 noisy / regime-shift / irregular 序列上的 routing 与 refinement。


## 8.5 RL 采样策略

每个阶段内至少按以下维度分桶：

- `reference_teacher_error_band`
- `normalized_permutation_entropy_band`
- `difficulty_stage`
- `offline_best_model`（辅助平衡用）

目的：

- 对齐论文式 curriculum；
- 避免 batch 长期被单一 easy 模式主导；
- 缓解单模型塌缩。


## 8.6 repair 样本的角色

repair 样本不是论文里的主动作空间，但可作为工程增强，用于：

- 第二轮 SFT 增量蒸馏；
- hard bucket 细化；
- failure analysis。

典型 repair 样本：

- 选错模型；
- 该 refine 却没有 refine；
- final answer 与 selected forecast 偏离过大；
- 长度/格式失败；
- 冗余工具调用。


## 9. 还需要补的工程约束

下面这些不是论文正文直接写出的机制，但对于当前代码库是必要的：

1. 保留格式解析与 stop string，避免 over-generation。
2. 加强 `Turn 3` prompt，强调 final answer 必须基于 selected forecast，而不是脱离 tool evidence 大幅重写。
3. 记录如下监控字段：
   - selected model
   - final answer 与 selected forecast 的距离
   - refine 前后 raw-scale MSE / MAE
   - format / length failure reason
4. 用这些 failure case 回流到下一轮 repair 数据。

这些都属于“工程稳定化”，不改变论文主线。


## 10. 需要修改的代码方向

## 10.1 agent flow

## 10.2 reward

需要改：

- `recipe/time_series_forecast/reward.py`

改动目标：

- 恢复论文式 composite reward；
- 采用 normalized + log-transformed MSE 主项；

- 保留 format validity 与 output length consistency；

  - state-dependent tool selection

## 10.4 RL 数据生成

需要改：

改动目标：

- 保持在线 prompt 简洁；

- 建立论文式 curriculum；
- 建立按 difficulty bands 的分桶采样。


## 11. 验收指标

新的验收重点应包括：

1. routing 指标
   - selected model distribution
   - single-model collapse ratio
   - 各 difficulty bucket 上的 routing 行为

2. forecasting 指标
   - raw-scale `MSE`
   - raw-scale `MAE`
   - normalized reward trend

3. refinement 指标
   - refine 触发率
   - refine 前后误差变化
   - final answer 与 selected forecast 的偏离分布

4. workflow 指标
   - 平均工具调用数
   - 冗余工具调用率
   - format / length failure 率


## 12. 最终决策

本版方案最终定为：

1. 严格保留论文主线：
   - `Turn 1` 诊断
   - `Turn 2` 单模型 routing
   - `Turn 3` agent-side refinement 与 final output

2. routing 仍由 RL 从 composite reward 中学出，不做显式工程增强监督。

3. reward 回到论文式 composite design。

4. SFT / RL 样本按论文逻辑重构。

5. 当前工程上的格式与提前收尾问题，通过：
   - 更强的 stage-aware prompt
   - 更合理的 reward
   - 更好的样本构造
   - 更完整的 failure logging
   来解决，而不是通过把最终 forecast 完全交给程序代写。
