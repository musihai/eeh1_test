# Cast-R1（ETTh1-OT）当前代码最佳修改方案（落地版）

## 1. 先给结论

基于你当前仓库代码、`debug_logs/final_launch_cmd.txt`、以及你上传的 `eval_step_aggregate.jsonl` / `eval_step_samples.jsonl`，**最好的修改路线不是继续优先折腾 parser，也不是先大改工具库，而是做一套“终止判定—reward—日志口径—训练日程”完全对齐的闭环**。

我建议你按下面的顺序改：

1. **先把流程层 final answer 判定改成“必须恰好 96 点”**，不再用当前 `>=8` 的宽松门槛。
2. **把 reward 改成“长度严格门控 + 原始口径 MSE 主优化”**，彻底消灭“95 点但前 95 点不错也能拿正奖励”的错误激励。
3. **把日志拆成原始误差口径和归一化误差口径两套**，避免后续调试时继续混淆。
4. **把 prompt 里的模型偏置去掉**，避免 agent 继续几乎固定塌缩到 `patchtst`。
5. **重新生成 teacher200、重跑 SFT，再拉长 RL**；当前 `20` 个 training steps 只能算链路测试，不能算真正训练。

## 2. 当前问题的核心判断

### 2.1 现在已经不是“完全不会生成答案”

从 `eval_step_aggregate.jsonl` 看：

- 总样本数：`1722`
- `pred_len == 96` 比例：`0.9814`，约 **1690 条**
- `pred_len in {94,95}` 比例：`0.00929`，约 **16 条**
- `pred_len < 96` 比例：`0.01626`，约 **28 条**
- `pred_len > 96` 比例：`0.00232`，约 **4 条**
- `missing_answer_close_tag_count = 3`
- `was_clipped_count = 3`

这说明：

- 格式整体已经比较稳定；
- 现在主问题不是“大面积格式崩坏”；
- 而是**少量但顽固的 95/94 点近完整错误**仍然被系统纵容。

### 2.2 你当前系统把“95 点答案”当成了可盈利策略

当前 `reward.py` 的逻辑是：

- 长度奖励：`len(pred) >= len(gt)` 直接给 `0.1`
- 长度惩罚：`len_gap == 1` 只罚 `0.08`
- MSE：按 `min(pred_len, gt_len)` 对齐后计算，再经 `1 / (1 + log(1 + mse))` 压缩，最后乘 `0.6`

这会导致一个结果：

- 95 点答案虽然长度不对；
- 但如果前 95 点写得还可以；
- 最终 reward 仍可能是正的，甚至不低。

例如你上传样例里：

- `pred_len=95, raw_mse=0.700007`，按当前 reward 估算，最终分数仍大约 **0.41**；
- `pred_len=95, raw_mse=1.828861`，最终分数仍大约 **0.31**。

所以从 RL 的角度，它会学到：

> “少写 1 个值，但前面写得不错，也还是赚的。”

这就是当前最关键的问题。

### 2.3 流程层和 reward 层标准不一致

你当前 `time_series_forecast_agent_flow.py` 里：

- `_expected_min_prediction_count()` 返回的是 `8`
- `_looks_like_forecast_answer()` 只要提取出的数值个数 `>= 8` 就认为“像预测答案”

这会带来一个明显后果：

- 流程层会很早把一个半成品当作 final answer 接受；
- reward 层再去扣长度罚分；
- 最终形成“流程接受半成品、reward 继续给正分”的错配。

### 2.4 你现在监控的 MSE 和 reward 实际优化的 MSE 不是同一件事

当前仓库里有两套口径：

1. `reward.py` 中：
   - 先用 **ground truth 的均值和方差**做归一化；
   - 再算 `raw_mse/raw_mae`；
   - 再做 log 压缩变成 reward。

2. `time_series_forecast_agent_flow.py` 中：
   - 用原始数值直接算 `MSE_data / MAE_data`；
   - 只是作为额外日志记录。

所以你现在经常会出现这种现象：

- 你以为自己在优化“原始 MSE/MAE”；
- 但训练真正优化的是“归一化后再压缩”的 reward。

这会直接影响你对实验结果的判断。

### 2.5 模型选择已经出现明显偏置

你上传的 `eval_step_samples.jsonl` 这 30 条样例切片里：

- `patchtst = 29`
- `itransformer = 1`

而且 `prompts.py` 中明确写着：

- `patchtst`: “Usually strong for ETTh1-like seasonal univariate patterns”
- `itransformer`: “usually less preferred”
- tool schema 里又写了 “Prefer 'patchtst' or 'arima'”

同时 `build_etth1_high_quality_sft.py` 又会把 reward 最优的 `best_model` 写成 `teacher_prediction_text`，作为 teacher 参考。

这意味着：

- prompt 在推 agent 选 `patchtst`；
- SFT 数据也在推 agent 模仿 `patchtst`；
- RL 在这基础上更容易继续塌缩到单模型。

### 2.6 训练步数太短，当前结果不能代表真正收敛结果

`debug_logs/final_launch_cmd.txt` 显示你这次 RL 配置里：

- `trainer.total_training_steps=20`
- `trainer.total_epochs=1`
- `actor_rollout_ref.rollout.n=2`
- `actor_rollout_ref.rollout.temperature=0.4`

这个训练量更像 smoke test，不像真正的 RL 收敛训练。

所以当前日志更适合用来**定位机制错误**，不适合用来判断“这个方法到底行不行”。

## 3. 本次修改的总体原则

本次不要同时大改太多地方。最优策略是：

### 原则 A：先解决“错误激励”

先把 95/94 点不再当作 profitable strategy。

### 原则 B：流程层和 reward 层必须使用同一标准

必须都是：

- 格式正确；
- `<answer>` 闭合；
- **恰好 96 个值**；
- 再谈 MSE/MAE。

### 原则 C：训练优化目标和你看的日志目标尽量一致

否则后面你调一个月，看到的指标和模型实际学的还是两码事。

### 原则 D：一次只引入一个主要优化目标

我建议第一版先做：

- **长度严格门控**
- **原始口径 MSE 主优化**
- **MAE 只记录不进 reward**

等稳定后，再决定要不要把 MAE 作为辅助 reward 加进去。

## 4.1 第一优先级：修改 `time_series_forecast_agent_flow.py`

### 目标

让流程层只接受“真正可用于 reward 的 final answer”，不再把 8 个以上数字的半成品当终答案。

### 必改 1：把 `_expected_min_prediction_count()` 改成严格 horizon

当前：

```python
def _expected_min_prediction_count(self) -> int:
    return 8
```

改成：

```python
def _expected_prediction_count(self) -> int:
    return int(self.forecast_horizon or 96)
```

并删除/废弃 `_expected_min_prediction_count()` 的宽松语义。

### 必改 2：把 `_looks_like_forecast_answer()` 改成“必须恰好 96 行纯数值”

建议改成：

```python
def _looks_like_forecast_answer(self, answer_text: Optional[str]) -> bool:
    if not answer_text:
        return False

    expected = self._expected_prediction_count()
    lines = [ln.strip() for ln in answer_text.splitlines() if ln.strip()]
    values = extract_values_from_time_series_string(answer_text)

    if len(lines) != expected:
        return False
    if len(values) != expected:
        return False

    for ln in lines:
        if re.match(r"^-?\d+\.?\d*$", ln) is None:
            return False

    return True
```

### 必改 3：final answer 提取失败时记录拒绝原因

建议在 `_extract_final_answer()` 里补一个 reject reason，例如：

```python
self.final_answer_reject_reason = None

...
if match:
    candidate = match.group(1).strip()
    if self._looks_like_forecast_answer(candidate):
        return candidate, 0.0
    self.final_answer_reject_reason = f"invalid_answer_shape:{len(extract_values_from_time_series_string(candidate))}"
    return None, 0.0
```

这样后面 debug 时可以直接看：

- 是没闭合；
- 还是 95 点；
- 还是混进了 prose。

### 必改 4：日志里同时记录原始口径和归一化口径指标

当前只记录：

- `MSE`
- `MAE`

建议改成同时记录：

```python
reward_extra_info.update({
    "orig_mse": orig_mse,
    "orig_mae": orig_mae,
    "norm_mse": norm_mse,
    "norm_mae": norm_mae,
    "pred_len": len(pred_values),
    "expected_len": len(gt_values),
    "final_answer_reject_reason": self.final_answer_reject_reason,
    "prediction_model_used": self.prediction_model_used,
    "selected_model": selected_model,
})
```

### 建议改 5：把工具进度奖励从 `0.05` 降到 `0.01` 或直接 `0.0`

当前：

```python
reward_score = 0.05 if tool_calls else 0.0
```

建议先改成：

```python
reward_score = 0.01 if tool_calls else 0.0
```

原因：

- 现在 workflow guard 已经能约束“不调用工具不行”；
- 你当前更关心的是最终预测质量；
- 太高的过程奖励会放大“流程看起来对”而不是“预测真的好”的倾向。

如果你后面发现模型开始偷懒不调工具，再考虑恢复到 `0.02`。

## 4.2 第二优先级：重写 `reward.py` 的长度门控与误差主目标

### 目标

让 reward 明确表达：

> “先长度完全正确，再比较预测误差；长度不对，哪怕前 95 点很准，也不能拿正奖励。”

### 必改 1：增加严格长度门控开关

建议在文件顶部增加：

```python
STRICT_LENGTH_GATE = os.getenv("TS_STRICT_LENGTH_GATE", "1").lower() in {"1", "true", "yes", "on"}
REWARD_USE_ORIG_SCALE = os.getenv("TS_REWARD_USE_ORIG_SCALE", "1").lower() in {"1", "true", "yes", "on"}
```

### 必改 2：把 `compute_length_score()` 改成“只有 exact 才给分”

当前逻辑：

- `len(pred) >= len(gt)` 就直接给 `0.1`

建议改成：

```python
def compute_length_score(solution_str: str, ground_truth: str) -> float:
    gt_values = extract_ground_truth_values(ground_truth)
    pred_values = extract_values_from_time_series_string(solution_str)
    if not gt_values:
        return 0.0
    return 0.1 if len(pred_values) == len(gt_values) else 0.0
```

### 必改 3：长度不匹配时，直接进入 hard gate，不再给 MSE 正奖励

这是这次最关键的改动。

建议在 `compute_score()` 里改成：

```python
length_hard_fail = False

if format_score >= 0 and gt_len > 0 and pred_len != gt_len and STRICT_LENGTH_GATE:
    length_hard_fail = True
    len_gap = abs(pred_len - gt_len)
    score = max(-1.0, -0.5 - 0.05 * min(len_gap, 10))
    mse_score = 0.0
```

这样：

- `95 != 96` 时不会再拿到正分；
- 但又不会和完全空输出一样粗暴；
- 还能保留长度偏差大小的信息。

### 必改 4：reward 的主误差项改成“原始口径 MSE”

当前 `reward.py` 里主 reward 用的是归一化 MSE。

考虑到你现在最想看的就是原始 MSE/MAE，而且数据集是 ETTh1 单变量 OT，**我建议第一版直接切换到原始口径 MSE 作为主优化目标**：

```python
orig_mse = float(np.mean((np.asarray(pred_values) - np.asarray(gt_values)) ** 2))
orig_mae = float(np.mean(np.abs(np.asarray(pred_values) - np.asarray(gt_values))))

norm_pred, norm_gt = normalize_for_reward(pred_values, gt_values)
norm_mse = float(mean_squared_error(norm_gt, norm_pred))
norm_mae = float(np.mean(np.abs(np.asarray(norm_gt) - np.asarray(norm_pred))))

main_mse = orig_mse if REWARD_USE_ORIG_SCALE else norm_mse
mse_score = (1.0 / (1.0 + np.log1p(main_mse))) * 0.6
```

### 必改 5：把 `raw_mse/raw_mae` 改名或补充为双口径日志

建议不要再让 `raw_mse` 这个名字表示“归一化后的 MSE”，因为很容易误导。

建议统一输出：

```python
"orig_mse": orig_mse,
"orig_mae": orig_mae,
"norm_mse": norm_mse,
"norm_mae": norm_mae,
```

并保留：

```python
"length_hard_fail": length_hard_fail,
"strict_length_match": pred_len == gt_len,
```

### 推荐的最终 reward 结构

我建议第一版就用下面这个结构：

- 格式错误：`-1.0`
- 长度错误：`[-1.0, -0.5]` 区间的 hard fail
- 长度正确：
  - exact bonus：`0.1`
  - MSE 主奖励：`0.6 / (1 + log1p(orig_mse))`
  - 暂时不加 MAE reward

也就是：

```python
if format_bad:
    score = -1.0
elif strict_length_gate and pred_len != gt_len:
    score = max(-1.0, -0.5 - 0.05 * min(len_gap, 10))
else:
    score = 0.1 + 0.6 / (1 + np.log1p(orig_mse))
```

这样最清楚，也最方便做归因。

## 4.3 第三优先级：修改 `prompts.py`，去掉显式模型偏置

### 目标

避免 agent 在 prompt 层面被硬推向 `patchtst`。

### 当前问题

你现在的 prompt 明确在暗示：

- `patchtst` 更优
- `itransformer` 不太推荐
- schema 里还写着 “Prefer patchtst or arima”

这会导致模型选择很难真正学出来。

### 建议改法

把描述从“偏好某模型”改成“根据窗口特征选择模型”。

#### 当前写法（不建议保留）

```python
- 'patchtst': Usually strong for ETTh1-like seasonal univariate patterns
- 'itransformer': Available, but usually less preferred for single-variable inputs
```

#### 建议写法

```python
- 'patchtst': useful when the window is smooth and shows strong local periodicity
- 'itransformer': useful when the window has stronger regime changes or long-range dependencies
- 'arima': useful when trend/seasonality is stable and mostly linear
- 'chronos2': useful fallback for irregular or noisy windows
```

tool schema 里也改成：

```python
"description": "Model to use. Choose based on the extracted features rather than a fixed preference."
```

### Turn 3 的 finishing 约束保留并强化

这部分你现在其实已经写得比较好了，不需要大改，只需要继续保留：

- `<answer>` 必须 EXACTLY 96 lines
- stop after the 96th value
- immediately output `</answer>`

## 4.4 第四优先级：处理 `build_etth1_high_quality_sft.py`

### 目标

让 teacher 选择目标和新的 reward 目标保持一致。

### 必须做的事

因为这个脚本直接 `from recipe.time_series_forecast.reward import compute_score`，所以：

> **只要你改了 reward，teacher200 就必须重新生成。**

否则会出现：

- teacher 数据按旧 reward 选出来；
- RL 按新 reward 优化；
- SFT warm start 和 RL 目标不一致。

### 当前建议

第一轮先不要强行做人为模型配额。

先做：

1. reward 改完；
2. 重新生成 teacher200；
3. 观察 `best_model` 分布是否仍极度塌缩。

### 如果仍然出现单模型塌缩，再做下面这个增强版

在 `select_curated_evaluations()` 里加一个可选的 model cap，例如：

- 任一模型在 curated teacher 中占比不超过 `70%`
- 剩余样本按 `selection_score` 补齐

但这一步我建议**放到第二轮**，不要一上来就加，不然你分不清性能变化到底来自 reward 还是来自 teacher 重采样。

## 8. 我最推荐你直接落地的“第一版代码策略”

如果你现在只想要一个**最稳、最不容易把系统改乱、同时最能解决当前问题**的版本，那就执行这一版：

### 第一版只做 4 件事

#### A. `agent_flow`
- final answer 必须 exact 96
- 记录 reject reason

#### B. `reward`
- 长度不等于 96 时直接 hard gate，不给正奖励
- 主 reward 改成原始口径 MSE

#### C. `prompts`
- 去掉 `patchtst preferred / itransformer less preferred`

#### D. 训练
- 重建 teacher200
- 重跑 SFT
- RL 至少跑到 `120 steps`

这是当前最优先、收益最大、风险最小的方案。

## 9. 不建议你现在做的事

下面这些我建议先不要做：

### 9.1 不要先重写 parser

当前主问题已经不是 parser。

### 9.2 不要一上来就把 MSE、MAE、趋势、拐点全塞回 reward

变量太多，后面根本不知道哪个改动起作用。

### 9.3 不要在 reward 改完之前继续直接长跑 RL

因为现在目标函数本身就有错误激励，继续跑只会把坏策略学得更稳。

### 9.4 不要在第一轮就强行做 teacher 多样性配额

先把长度和 reward 机制修好，再看模型分布是否还需要人为平衡。

## 10. 最终一句话总结

**你当前最应该做的，不是继续让模型“更会预测”，而是先让训练系统不再奖励错误行为。**

具体来说就是：

> **把 final answer 判定改严格，把 reward 改成“长度严格门控 + 原始口径 MSE 主优化”，把日志口径拆清楚，再重建 teacher200/SFT/RL。**

这套修改完成后，你的 RL 才是在朝着你真正想要的目标学，而不是在学“95 点但前 95 点不错也没关系”。
